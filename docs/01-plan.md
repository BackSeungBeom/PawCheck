# 01. 기획서 — PawCheck

## 0. 공모전 연계 정보

- 공모전: 2026 관광데이터 활용 공모전 ②-2 웹·앱 구현 부문
- 지정과제: **과제 6 — 모호한 반려동물 출입 조건 및 규정 인지 오류로 현장 입장 거부 및
  헛걸음 유발 문제**
- 필수 요건: 한국관광공사 OpenAPI(KorPetTourService2) 필수 활용 + 지정과제 문제 해소 방향 서비스 기획
- 심사 배점(1차, 100점): 서비스 구현성 30 / 기획력 30 / 데이터 활용 적절성 20 / 발전성 20
  → 본 기획서·설계서는 이 네 항목을 모두 겨냥해 작성됨

## 1. 문제 정의

### 1-1. 표면적 문제
반려동물 동반 가능 시설이 늘고 있지만, 실제 출입 규정은 시설마다 다르고 정보가 SNS·블로그·
지도 서비스에 흩어져 있거나 최신이 아니어서, 현장에 도착한 뒤 입장을 거부당하거나 헛걸음을
하는 문제가 반복된다.

### 1-2. 대표 사례
- "반려동물 가능"이라 되어 있지만 실내는 불가능
- 맹견은 입마개 착용 필수 등 견종별 조건
- 목줄/캐리어 등 동반 시 필요사항 불명확
- 시설 내 일부 구역(카페 내부, 미술관, 캠핑장 취사장 등)만 예외적으로 불가

### 1-3. 핵심 문제 정의
> 시설별 반려동물 출입 규정이 표준화되어 있지 않고, 예외조항이 자연어 형태로 흩어져 있어
> 사용자가 출발 전에 실제 출입 가능 여부를 판단하기 어렵다.

## 2. 실측 데이터 기반 재설계 (중요)

기획 초안에서는 "12kg 이하", "이동가방 필수 O/X" 같은 정량적 조건을 자동 판별하는 것을
목표로 했으나, 한국관광공사 KorPetTourService2 API를 실제로 호출해 172건을 표본 조사한 결과:

- 체중(kg) 기반 제한 필드는 **API에 존재하지 않음** → 이 조건의 자동판별은 기획에서 제외
- 반면 `acmpyTypeCd`(동반유형), `acmpyPsblCpam`(견종제한), `acmpyNeedMtr`(필요사항)는
  관광지 기준 93~100% 채워져 있고, 표현이 몇 가지 패턴으로 정형화되어 있어 규칙 기반 파싱이
  가능함이 확인됨
- `etcAcmpyInfo`(기타정보)에는 "실내 미술관은 동반 불가", "캠핑장은 동반불가" 같은
  **시설 내 예외조항**이 자유문장으로 들어있어, 이 부분만 LLM으로 구조화하는 것이 효과적임
  (상세 근거는 `03-data-spec.md` 참고)

→ **기획을 "규칙+AI 하이브리드 파서" 구조로 재정의.** 이는 단순 API 연동이 아니라
"비정형 공공데이터를 실사용 가능한 정보로 가공"하는 기술적 기여이므로, 데이터 활용
적절성(20점) 항목에서 설득력 있는 근거가 됨.

## 3. 타깃 사용자

반려동물과 함께 국내 여행·나들이를 하려는 반려가구. 특히:
- 처음 가보는 지역/시설이라 정보를 사전에 확인해야 하는 자유여행객
- 맹견/대형견 보호자 등 출입 제한에 민감한 사용자
- "가서 안 되면 어쩌지" 하는 불안감 때문에 외출 자체를 꺼리는 사용자

## 4. 서비스명 및 한 문장 정의

**서비스명: PawCheck**

> 반려동물 출입 규정을 표준화하고, 반려동물의 조건과 시설 규정을 자동으로 비교하여
> 출발 전에 실제 입장 가능 여부를 예측해 헛걸음과 현장 입장 거부를 줄이는 웹 서비스

## 5. 핵심 기능

### 5-1. 출입 조건 체크리스트 변환
공사 API 원문(자연어)을 아래와 같은 표준 체크리스트로 변환해 보여준다.

| 항목 | 표시 |
|---|---|
| 실내 동반 가능 | O / X / 조건부 |
| 실외 동반 가능 | O / X / 조건부 |
| 맹견 제한 | O / X (제한 시 조건 명시) |
| 목줄 착용 필수 | O / X |
| 입마개 필요(맹견 한정 등) | O / X |
| 시설 내 예외구역 | 있음(목록) / 없음 |

### 5-2. 내 반려동물 기준 맞춤 판정
사용자가 자신의 반려동물 정보(견종/크기, 맹견 해당 여부, 목줄·캐리어 소지 여부)를 입력하면,
체크리스트와 비교해 아래 3단계로 안내한다.
- ✅ 입장 가능
- ⚠️ 조건부 가능 (예: "맹견은 입마개 착용 시 가능" — 사용자가 준비했는지 확인)
- ❌ 정보상 불가 / 정보 부족 (자동판정 불가 시 원문 그대로 노출 + "현장 확인 권장" 안내)

주의: API 데이터가 100% 완전하지 않으므로, "무조건 O/X 단정"이 아니라 **"현재까지 확인된
정보 기준"**이라는 톤을 UI 전체에 일관되게 유지한다 (오정보로 인한 책임 문제 예방).

### 5-3. 시설별 규정 비교
같은 지역/카테고리 내 여러 시설의 체크리스트를 나란히 비교할 수 있는 뷰 제공.

### 5-4. 커뮤니티 검증
최근 방문자가 "규정 그대로였는지" 확인/갱신하는 기능. 방문일 + 일치여부(O/X) + 코멘트로
구성하며, 최신순 가중치를 반영한 신뢰도 지수를 함께 표시한다. (콜드스타트 대응은
`02-design.md` 참고)

### 5-5. 출발 전 체크리스트
공사 API의 `relaFrnshPrdlst`(비치품목)/`relaRntlPrdlst`(렌탈품목) 데이터를 활용해
"이 시설은 O을 제공/대여합니다" + "이건 직접 챙기세요"를 자동 안내.

## 6. 차별점

| 기존 지도/블로그 서비스 | PawCheck |
|---|---|
| "반려동물 가능" 여부만 단편 표시 | 출입 조건을 항목별로 구조화해 표시 |
| 후기를 사람이 직접 찾아 읽어야 함 | 내 반려동물 조건에 맞춰 자동 판정 |
| 정보 최신성 확인 불가 | 최근 방문자 검증 + 신뢰도 지수 |
| 예외조항(부분 불가 구역 등) 누락되기 쉬움 | LLM으로 예외조항까지 구조화해 노출 |

## 7. MVP 범위 (실측 데이터 기준 우선순위)

| 카테고리 | contentTypeId | 실측 데이터 품질 | MVP 포함 여부 |
|---|---|---|---|
| 관광지 | 12 | 핵심필드 93~100% | ✅ 1순위 |
| 레포츠 | 28 | 핵심필드 73~93%, 구비시설 정보 상대적으로 풍부 | ✅ 1순위 |
| 숙박 | 32 | 73~80% (예외조항 정보는 얕음) | 2순위 (확장) |
| 음식점 | 39 | 30~67% | 2순위 (확장) |
| 쇼핑 | 38 | 사실상 데이터 없음(핵심 필드 0%) | ❌ 제외 |
| 축제/공연 | 15 | 표본 자체 미미(3건) | ❌ 제외 |

MVP는 **관광지 + 레포츠** 두 카테고리로 완성도를 확보하고, 로드맵 후반부에 숙박/음식점으로
확장 가능한 구조로 설계한다 (`02-design.md`의 데이터 파이프라인이 카테고리 무관하게 재사용
가능하도록 설계됨).

## 8. 기대 효과

- 헛걸음으로 인한 반려인의 시간·교통비 손실 감소
- 시설과 이용자 간 현장 갈등 감소
- 방문자 검증 기반의 최신 규정 공유 문화 형성
- 반려동물 동반 외출에 대한 심리적 진입장벽 완화

## 9. 발전 가능성 (심사기준 "서비스 발전성" 대응)

- 카테고리 확장: 숙박 → 음식점 → 쇼핑(데이터 개선 시)
- 다른 공사 API(관광지 연관정보, 지역 축제 등)와 연계해 "반려동물 동반 가능 코스" 추천으로 확장
- 커뮤니티 검증 데이터가 쌓이면 시설 측에 규정 정비를 위한 피드백 리포트 제공 가능
- 콜드스타트 해소 전략: 초기에는 팀이 직접 방문조사한 시드 데이터 일부를 커뮤니티 후기에
  선반영하고, 이후 실사용자 기여로 자연스럽게 전환
